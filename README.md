## Decisions & Tradeoffs

### Transform (Part 2)

**Field name normalization**  
The raw data uses three distinct schema shapes (`case_no`/`CaseNumber`/`permit_id`, etc.). I used a `pick(record, ...keys)` helper that tries each candidate key in order and returns the first non-null, non-empty value. This avoids a fragile key-remapping object and handles the null/empty-string edge cases cleanly.

**Deduplication — address as the key**  
The spec says "same address = same record." My `addrKey()` function lowercases, strips unit/suite suffixes, collapses whitespace, removes trailing punctuation, and normalizes common abbreviations (Street→St, Boulevard→Blvd, etc.) before comparing. This correctly merges:
- `"2847 Riverside Dr"` + `"2847  Riverside Dr."` → same project
- `"445 Elm Street"` + `"445 elm st"` → same building
- `"3321 Belmont Blvd"` + `"3321 BELMONT BLVD"` → same residence
- `"1005 Monroe St"` + `"1005 Monroe St, Unit B"` → same address (unit stripped)

**Merge conflict resolution**  
When two records share an address and both have a value for the same field, I take the value from whichever record has the more recent `application_date` (per spec). The trade-off: for `1005 Monroe St`, this merges a planning case ($4.2M townhomes) with an interior finishes permit ($125K) and the merged `estimated_cost` becomes $125K — clearly the sub-permit's scope, not the project's total value. In a production pipeline I'd keep the highest cost, or keep both costs with separate labels; for this exercise I followed the spec literally and flagged it.

**Records filtered out (2 removed)**  
- `RES-2024-10370`: `street_address` is `""` and `name` is `null` — no way to identify this record
- `PLN-2024-00216`: `location` is `null` and `project_title` is `null` — same issue

**Status and project_type normalization**  
Both use `Map` lookups on the lowercased raw value. For `project_type`, unrecognized values are kept as-is (per spec). For `status`, unrecognized values are set to `null` with the original preserved in `raw_data`.

**Cost parsing**  
Raw values mix strings with currency formatting (`"$24,500,000"`), plain numbers (`450000`), and string numbers (`"485000"`). A single strip-and-parse handles all three.

**Date parsing**  
Handled six formats: ISO, ISO with time, US numeric, US short (2-digit year), long-form month, and DMY-with-abbrev. All converted to `YYYY-MM-DD`.

### Extraction (Part 3)

**Prompt design**  
The system prompt establishes the role (data extraction specialist, pipeline context). The user prompt explicitly lists every output field with type, gives the valid role enum, and includes a concrete one-line example. Rules are stated positively ("only named individuals") to reduce hallucination.

**Unnamed firms**  
Terracon Consultants (doc_1) and Neel-Schaffer Inc. (doc_3) are mentioned but no individual is named — the prompt instructs the model to skip these, so they correctly don't appear in output.

**Superseded professional**  
Doc_3 mentions Tom Loss was originally assigned but has been replaced by Angela Wright. The prompt says "if a person appears to have been replaced, include the current assignee only" — Angela Wright is extracted, Tom Loss is not.

**JSON parsing resilience**  
The `parseJsonArray()` function first attempts a direct JSON parse; if that fails, it looks for the first `[` and last `]` in the response and retries. This handles the common LLM failure mode of wrapping JSON in prose or markdown code fences.

**professionals.json**  
The included `professionals.json` was hand-generated by applying the same extraction logic I encoded in the prompt. Running `extract.js` with a valid API key will produce equivalent output.

## AI Tools & Process

**What I used:**  
Claude Code (Anthropic's CLI) throughout — for researching Austin's portal stack, writing `transform.js`, designing the extraction prompt, and structuring this README.

**How I used it:**  
- Research: asked Claude to describe Austin's permit infrastructure, then cross-checked URLs and platform indicators against what I'd verify in a browser's Network tab
- Code generation: generated the initial `transform.js` skeleton, then manually reviewed every normalization map and edge-case branch against the raw data
- Prompt iteration: drafted the extraction prompt, then walked through each document mentally to check the rules would produce correct output (e.g., the "superseded professional" rule for doc_3's Tom Loss / Angela Wright situation)

**Where I still had to apply my own judgment:**  
- Deciding that the `1005 Monroe St` merge (planning case + sub-permit) produces a misleading `estimated_cost` — Claude followed the spec; I flagged it as a known limitation
- Choosing Austin as the research city and deciding which of its four portals were worth documenting vs. skipping
- Deciding that Michelle Torres (the staff reviewer in doc_3) is a city employee, not a project professional, and should be excluded from extraction

## What I'd Improve

- **Dedup cost field**: keep the maximum `estimated_cost` when merging a planning case with sub-permits — the higher value almost always represents total project scope
- **Address geocoding**: some records have no lat/lng; in production I'd run a geocoder (Google Maps or Census TIGER) over the `address` field after transform
- **Streaming for large batches**: `extract.js` processes documents sequentially; for production I'd use `Promise.allSettled` with a concurrency limiter (~5 parallel) and add retry logic with exponential backoff
- **Prompt caching**: the system prompt and instructions are static — adding Anthropic cache-control headers on the long system prompt would cut API cost ~90% on large batches
- **AB+C scraping**: the Research section documents that Accela has no public API. I'd build a Playwright-based scraper with session management and rate limiting, writing to a staging table before the transform step
- **Status catch-all logging**: unknown status values are silently set to `null`; in production I'd emit these to a monitoring dashboard so new status strings get mapped quickly
